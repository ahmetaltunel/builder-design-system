## Project Overview

BuilderDesignSystem is a desktop-first macOS SwiftUI design-system package for calm, utility-focused builder tools. The repository is organized as a layered system, not as a demo-first app. Treat the design system as the source of truth and the showcase as a consumer that exercises that truth.

## Architecture

### Core runtime layers

- `Sources/BuilderFoundation/`
  - Semantic token enums
  - Theme and material resolution
  - Density, contrast, visual context, and mode types
  - Typography, shadow, motion, icon, and grid specs
- `Sources/BuilderComponents/`
  - Reusable SwiftUI surfaces, controls, navigation, forms, feedback, and content primitives
- `Sources/BuilderDesignSystem/`
  - Public umbrella module that re-exports `BuilderFoundation` and `BuilderComponents`

### Support and delivery layers

- `Sources/BuilderReferenceExamples/`
  - Canonical SwiftUI examples used by generated docs and showcase previews
- `Sources/BuilderCatalog/`
  - Foundation, component, and pattern taxonomy plus cross-reference metadata
- `Sources/BuilderShowcase/`
  - Executable app for demos, manual QA, and inspection
- `Sources/BuilderArtifactGenerator/`
  - Regenerates token code, exports, and reference docs from the manifest and catalog

### Source-of-truth rules

- Put tokens, theme rules, and shared styling contracts in `BuilderFoundation`.
- Put reusable view behavior and appearance in `BuilderComponents`.
- Keep `BuilderShowcase` as a consumer. Do not let showcase-only styling, copy, or route structure become the system contract.
- Treat `BuilderReferenceExamples` as the canonical example layer for docs and previews.
- Treat `BuilderCatalog` as structured metadata that explains the system; do not turn it into product logic.

## Generated Artifacts

Several files are generated and should be changed through their source inputs, not hand-edited as the primary workflow:

- `DesignTokens/design-token-manifest.json` is the source of truth for token definitions.
- `Sources/BuilderFoundation/Tokens/*.swift` and `Sources/BuilderFoundation/Generated/DesignTokenRegistry.generated.swift` are regenerated by `BuilderArtifactGenerator`.
- `Exports/design-tokens.json` and `Docs/System/TokenReference.md` are regenerated from the token manifest.
- `Docs/Foundations/`, `Docs/Components/`, and `Docs/Patterns/` are regenerated from catalog metadata and reference examples.

If a change affects tokens, catalog metadata, examples, or generated docs, run the generator-driven validation flow before considering the work complete.

## Finding Related Code

1. Start from the layer that should own the change.
2. For component work, inspect all of these together:
   - `Sources/BuilderComponents/`
   - `Sources/BuilderReferenceExamples/`
   - `Sources/BuilderCatalog/Components/`
   - `Docs/Components/`
3. For foundation work, inspect:
   - `DesignTokens/design-token-manifest.json`
   - `Sources/BuilderFoundation/`
   - `Sources/BuilderCatalog/Foundations/`
   - `Docs/Foundations/`
4. For pattern work, inspect:
   - `Sources/BuilderReferenceExamples/`
   - `Sources/BuilderCatalog/Patterns/`
   - `Docs/Patterns/`
5. Use `Sources/BuilderShowcase/` to understand how the system is exercised in real screens, not to define the contract.
6. Check `Tests/BuilderComponentsTests/`, `Tests/BuilderCatalogTests/`, and `Tests/BuilderIntegrationTests/` for expected behavior and snapshot coverage.

## Validating Changes

Mandatory: run the repository validation script before declaring work complete for code, token, catalog, example, or generated-doc changes.

```bash
Scripts/validate-design-system.sh
```

That script:

- runs `swift run BuilderArtifactGenerator`
- runs `swift build`
- runs `swift test`

Additional validation guidance:

- If the change affects visuals or interaction states, also inspect the result in the showcase with `swift run BuilderShowcase`.
- Validate affected screens in light and dark themes.
- Validate compact, default, and comfortable density behavior when layout or controls change.
- Do not leave generated outputs stale after changing their source inputs.

## Coding Guidelines

### Design-system expectations

- Build from semantic tokens before adding view-local styling.
- Reuse existing components, specs, and token roles before introducing new ones.
- Preserve the calibrated visual intent: quiet, precise, dense-but-readable, native to macOS, restrained in color and motion.
- Prefer system-wide contracts over one-off fixes in showcase code.
- Keep public consumer entrypoints centered on `BuilderDesignSystem`.

### Component and pattern work

- New components should ship with catalog metadata, a canonical reference example, accessibility guidance, density behavior, theming hooks, and representative test coverage.
- New patterns should include when-to-use guidance, required components, layout recipe, content guidance, accessibility notes, and a representative example scaffold.
- If a demo needs a reusable pattern, promote it into the system layer first instead of hard-coding it in the showcase.

### Content and copy

- Follow `Docs/System/ContentGuidelines.md`.
- Use calm, operational, desktop-first language.
- Prefer direct labels and recovery guidance over vague or marketing-heavy copy.
- Keep terminology consistent across components, docs, catalog metadata, and showcase examples.

### Swift style

- Match the surrounding Swift style and naming conventions already used in the repository.
- Prefer small, composable SwiftUI views and keep public API additions intentional.
- Avoid hard-coded colors when a semantic token should own the decision.
- Use accessibility labels, hints, and states where component behavior needs them.

## Repo Learnings

- The showcase is intentionally not the source of truth. When in doubt, fix the system layer and let the showcase consume it.
- Snapshot coverage in `Tests/BuilderIntegrationTests/BuilderSnapshotTests.swift` is part of the quality bar for visual changes.
- Changes that affect design-system maturity should also consider updates to `CHANGELOG.md`, governance docs, or handbook content when appropriate.
